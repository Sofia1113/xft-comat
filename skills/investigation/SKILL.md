# 调查方法论（investigation）

遇到 bug 走「复现与根因」，准备重构走「行为基线」。共同纪律：**先取证，后断言；调查时不动手改代码。**

## 铁律（缺陷调查）

```
不先找到根因，就不提修复方案
```

随手改、打补丁只会掩盖问题、制造新 bug。症状级修复等于失败。没完成根因调查之前，你不能提修复方案。

## bugfix：四个阶段（按顺序完成）

### 阶段 1：根因调查

1. **仔细读错误信息**——别跳过；堆栈、行号、错误码常常直接指向答案，把堆栈读完整。
2. **稳定复现**——能可靠触发吗？确切步骤？每次都发生吗？不能复现就先收集数据，不要猜。UI 复现需要浏览器时用 agent-browser 取真实页面证据。
3. **检查近期变更**——git diff、最近提交、新依赖、配置或环境差异。
4. **多组件系统先埋点取证**——在每个组件边界记录进/出的数据与配置传播，先跑一次定位"在哪一层断的"，再深入那一层调查；不要一上来就猜。
5. **追数据流**——坏值从哪来？谁用坏值调了它？一路往上追到源头，在源头修，不在症状处修。

### 阶段 2：模式分析

找到相似的可用代码，**完整**读参考实现（别略读），逐条列出"可用"与"损坏"之间的差异（别假设"那个不可能有影响"），弄清它依赖什么、做了哪些假设。

### 阶段 3：假设与验证

形成**单一**假设（"我认为 X 是根因，因为 Y"），用最小改动验证它（一次只动一个变量），验证通过再继续。不懂就说不懂，不要假装。

### 阶段 4：实现修复

先写复现该 bug 的失败测试（见 tdd），再做**单一**修复（只针对根因，不顺手改别的），然后验证：原症状消失、没有测试被改坏。

**若修复无效：停。** 数一下已经试了几次。<3 次：回阶段 1，带着新信息重新分析。**≥3 次：停下质疑架构**——每次修复都牵出别处的新问题，往往说明是架构错了，而不是又一个失败的假设；此时与用户讨论后再动，不要直接尝试第 4 次修复。

## refactor：行为基线

1. **圈定不变约束**——明确哪些外部行为不能变（输出、API 契约、副作用、性能特征）。
2. **建立行为基线**——找到覆盖待改行为的现有测试并运行，记录当前全绿快照；缺保护的行为列为安全网缺口（待补保护测试）。
3. **识别风险热点**——指出重构中最易破坏行为等价的位置（隐式依赖、共享状态、顺序耦合）。

## 红旗——停下，回到取证

- "先快速修一下，回头再查"
- "随便改改 X 看看行不行"
- "一次多改几处，跑测试"
- "我大概知道是 X，直接修"
- 还没追数据流就开始报方案
- 已经试了 2+ 次还想"再修一次"
- 每次修复都在别处冒出新问题

## 常见借口

| 借口 | 现实 |
|------|------|
| "问题很简单，不用走流程" | 简单问题也有根因，流程对简单 bug 很快 |
| "紧急，没时间走流程" | 系统化调查比反复猜-试更快 |
| "先试这个，回头再查" | 第一次的修法定下基调，一开始就做对 |
| "我看到问题了，直接修" | 看到症状 ≠ 理解根因 |
| "再修一次"（已失败 2+ 次） | 3+ 次失败 = 架构问题，质疑模式而不是再修一次 |

## 边界

- 调查时不做大范围重构，不直接实现修复。
- 证据不足时只写假设并显式标注，不得断言根因；先换取证路径（换日志源、扩大时间窗、加临时日志复跑、换工具）再下结论。
- UI 问题需要真实浏览器证据，或显式说明环境不足。
