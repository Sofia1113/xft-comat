# 端到端验证方法论（E2E）

## 概述

宣称"完成/可用/通过"而没有新鲜的端到端证据，是不诚实，不是高效。

**核心原则：先有证据，再下结论。**

## 铁律

```
没有亲自跑出的端到端证据，就不宣称端到端可用
```

UI 任务缺少真实浏览器证据，就是没通过——单元测试绿、build 成功都不算。

## 方法

1. **转用户路径**：把测试用例与验收标准翻译成真实用户操作路径（打开什么、点什么、填什么、期望看到什么）。
2. **明确启动条件**：应用如何启动、URL、账号、测试数据、前置状态。条件不具备就显式记录为阻塞，不假装验证过。
3. **执行真实回归**：
   - **UI / 浏览器流程**：必须用 agent-browser 驱动真实浏览器，观察可访问性树与页面行为取证。
   - **非 UI**：跑端到端的真实链路（API → 服务 → 存储 等），不只跑单元测试。
4. **确认修复闭环**：核对此前审查发现确已修复，且修复未引入新回归。
5. **逐用例判定并决定是否新一轮**：逐用例记 通过/失败；若验收范围或失败假设变了，开新一轮验证。

## 何时应用

准备做出以下任何一种表述之前，都要先有新鲜证据：

- 任何形式的"完成/修复/通过/可用"。
- 任何"应该没问题""看起来对了"之类的满意表态。
- 收尾交付、宣布验收通过、转入下一阶段。

## 红旗——停下

- 用"应该""大概""看起来"来描述结果
- 还没跑验证就表达满意（"搞定了！""完美！"）
- UI 任务用单元测试或 build 成功来冒充 E2E
- 相信"实现者报告说成功了"而不独立核验
- 凭部分验证（跑了一个用例）推断整体通过
- 想着"就这一次先不验了"

## 常见借口

| 借口 | 现实 |
|------|------|
| "应该能用了" | 去跑验证 |
| "我很有把握" | 把握 ≠ 证据 |
| "build 过了" | build ≠ 端到端可用 |
| "单测都绿了" | 单测 ≠ 真实用户路径 |
| "实现者说修好了" | 独立核验 |
| "UI 我本地点过了" | 没有可复核的真实浏览器证据就是没通过 |

## 边界

- **不能用单元测试或 build 成功冒充 E2E**；UI 任务缺少真实浏览器证据就是没通过。
- 不用 E2E 替代代码审查（审查应已在更早环节做完），不写业务代码（失败回交实现者修复）。
- 必须发生在审查与修复闭环之后。

## 产出

- E2E 范围与启动条件。
- 审查后回归状态与 agent-browser 使用情况（UI 任务必须有真实操作证据）。
- 逐用例 通过/失败 结果。
- 失败项与下一轮建议。
