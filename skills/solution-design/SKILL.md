# 方案设计方法论（solution-design）

基于已确认的需求产出可落地方案、风险与推荐，帮用户做关键抉择。修复策略、重构方案同理，方法相同。

**核心原则：单一推荐 + 必要时的备选 + 显式标出需要用户拍板的取舍。** DRY、YAGNI、TDD。

## 方法

1. **单一推荐**：基于已确认需求提出 1 个推荐方案；只有存在**真实**架构取舍时才补充备选，最多 2 个。不为凑数硬造备选。
2. **多维比较**：比较方案的复杂度、风险、测试策略和迁移成本。
3. **理由可溯**：明确推荐方案和理由，并说明哪些需求澄清结果支撑该方案。
4. **标决策点**：把关键架构取舍列为「待用户拍板的决策点」，每个附推荐选项与理由。拍板由主导者向用户拿到后写回——你不替用户填"最终抉择"，未拍板前不得把方案写成已定稿。
5. **前端约束**：如果任务包含前端视觉或交互要求，明确需要前端设计参与的设计约束和验收点。
6. **YAGNI**：从方案里删掉需求没要求的功能、抽象层和"以后可能用得上"的灵活性。设计服务于当前目标，不堆技术。

## 设计为隔离与清晰服务

- 把系统拆成职责单一、通过清晰接口通信、能独立理解与测试的小单元。每个单元都能回答：它做什么、怎么用、依赖什么。
- 你对能一次装进上下文的代码推理得最好，文件聚焦时改动更可靠。优先小而专注的文件，而非什么都做的大文件。
- **一起变化的代码放在一起**：按职责切分，而不是按技术分层切分。
- 在既有代码库里：先探索现有结构与命名，沿用既有模式，不擅自大改；但你正在改的文件若已臃肿到失控，把"拆分"纳入设计是合理的。不要提与当前目标无关的重构。

## 边界

- 不写业务代码，不执行测试（实现交给后续环节）。
- 不替用户做产品取舍——拿不准时给推荐并让用户选。
- 困难任务不得把方案写成已定稿，用户抉择由主导者拿到后写回。

## 输出格式

```markdown
## 方案目标

## 推荐方案
- 做法：
- 理由（对应哪条需求澄清）：
- 风险：
- 测试策略：

## 备选方案（仅在存在真实取舍时填写）
### 方案 B：
- 做法：
- 适用条件：
- 主要权衡：

## 需要用户抉择
1. 无 / 决策点（附推荐选项与理由）
```

## 示例（节选，按实际任务改写，禁止保留占位）

```markdown
## 推荐方案
- 做法：权限用 RBAC，角色-权限多对多存数据库，前端按角色渲染勾选项
- 理由：需求要求"角色列表 + 权限勾选"，未来角色会增多，RBAC 扩展成本低
- 风险：权限缓存与实时回收一致性
- 测试策略：权限矩阵单测 + 越权访问回归 + E2E 勾选保存

## 备选方案
### 方案 B：直接在用户表加权限位图
- 适用条件：角色固定且 ≤8 种
- 主要权衡：实现快但扩展和审计差

## 需要用户抉择
1. 角色是否需要运行时自定义（影响选 RBAC 还是位图）
```

方案落定后，按 [[task-splitting]] 把实现拆成可并发的独立任务；业务规则复杂时先按 [[domain-modeling]] 建模再设计。
