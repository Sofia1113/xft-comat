# 想法拷问器（grill-idea）

替代「需求清单式」澄清：目标不是把表格填满，而是**还原用户脑中的想法**——他为什么要做这件事、想得到什么、心里已经有什么画面、还有哪些没说出口的假设。直到你能完整复述这个想法、用户确认无偏差，工作才算可以往下走。

本 skill 源自 grill 模式：**不留死角地持续访谈，沿决策树走完每一条相关分支，按依赖顺序逐个解决决策，直到达成共同理解。**

<硬门禁>
在你呈现出对想法的复述、且用户确认无偏差之前，不要写代码、不要搭脚手架、不要采取任何实现动作。无论需求看起来多简单，都适用。
</硬门禁>

## 反模式："这太简单了不用澄清"

每件事都走这个过程。"简单"恰恰是未经检验的假设最容易造成返工的地方。复述可以很短（简单需求几句话即可），但你必须呈现出来并得到确认。

## 核心机制（不可省略）

1. **一次只问一个问题**。问完等回答，回答决定下一个问题——不要把问题清单一次性倒给用户。
2. **每个问题附带你的推荐答案**和一句理由。优先用 AskUserQuestion 等运行时提问工具，推荐答案放第一个选项并标注「（推荐）」；工具不可用时才降级为下方文本格式。
3. **能通过探索代码库回答的问题，去探索代码库**，不要问用户：现有实现模式、框架与工具选择、相似路由/组件/API/测试、代码里已有的业务规则。探索所得记录为带文件证据的推断，只有后果有风险时才向用户确认。
4. **沿决策树走，按依赖顺序解决**：先落定影响面最大的主干决策，它的答案决定下一层要问什么；上一个决策未落定时，不开启依赖它的分支。
5. **持续追问直到达成共同理解**：完成的标准不是问满 N 轮，也不是清单打钩，而是「你能复述用户的想法，且用户认可没有走样」。
6. **YAGNI**：从想法里识别并剔除用户没要、只是"以后可能用得上"的功能，不要把范围越问越大。

## 决策树（主干 → 分支，按依赖顺序）

与当前任务无关的分支明确跳过，不为问而问：

1. **动机**：用户想解决什么问题？为什么是现在？——不理解动机就无法判断范围与取舍。
2. **期望成果**：做完后用户拿到什么？谁在用、用在哪个场景？
3. **形态与行为**：用户脑中已有的画面是什么——交互、流程、输入输出、数据从哪来到哪去？哪些是硬约束，哪些他愿意交给你决定？
4. **边界与例外**：明确不做什么（非目标）？边界情况和出错时期望什么行为？
5. **验收**：用户怎么判断「做对了」？哪些场景必须被验证？
6. **范围信号**：是否涉及 UI/浏览器流程、API/数据、权限/认证、迁移或跨模块变更——这些直接改变工作量与风险定级。

## 文本降级格式

仅在运行时提问工具不可用时使用：

```markdown
## 澄清

问题：<一个具体决策或缺失事实>

推荐答案：<你会选的默认答案和理由>

为什么重要：<该答案如何影响范围、设计、测试或验收>
```

## 边界

- 澄清充分前，不写最终需求，不进入实现。
- 不替用户拍板高风险产品取舍——拿不准时给推荐并让用户选。
- 访谈保持低延迟问答，不绕第三方转述。

## 完成标准（共同理解）

收尾前，向用户**复述一遍你理解的想法**：要什么、为什么、长什么样、不做什么、怎么算做成。用户确认无偏差，且满足：

- 决策树每条相关主干都有用户答案、代码证据或显式标记的假设。
- 已识别哪些回答改变了复杂度或风险。
- 是否涉及 UI/浏览器流程、API/数据、权限/认证、迁移或跨模块已有结论。

## 输出（交接给后续路由/设计）

```markdown
## 想法还原（已向用户复述并确认）

## 已确认决策
- 决策：影响

## 代码推断
- 事实：证据

## 剩余假设
- 假设：风险

## 范围信号
- 任务类型：
- 复杂度维度：
- UI/E2E：
```
