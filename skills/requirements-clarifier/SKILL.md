# 需求澄清器

在 xft-comat 路由前的 `explore-and-clarify` 阶段使用本 skill。目标是在运行 `workflowctl.ts route` 前，让任务足够清晰，能够被路由和实现。

本 skill 借鉴 `grill-me` 模式：通过持续追问建立共同理解，按依赖顺序遍历决策树，每次先解决一个决策，再开启下一个决策。

## 运行规则

- **优先使用运行时提问工具**（AskUserQuestion、request_user_input 或等价工具）：推荐答案设为第一个选项并标注推荐；只有工具不可用时才降级为下方 markdown 文本格式提问。
- 每次只问一个问题。
- 每个问题都附带你的推荐答案。
- 如果问题能通过探索代码库回答，就探索代码库，不要询问用户。
- 持续追问，直到目标、非目标、业务规则、边界情况、验收标准和影响路由的风险都足够明确。
- 澄清充分前，不创建 `.xft-comat`，不调用 `route`，不写最终需求。
- 如果某个决策对路由或实现并非必要，先延后，不要扩大访谈范围。

## 问题格式

面向用户的问题使用以下结构：

```markdown
## 澄清

问题：<一个具体决策或缺失事实>

推荐答案：<你会选择的默认答案和理由>

为什么重要：<该答案如何影响路由、范围、设计、测试或验收>
```

## 先探索代码

提问前，先检查本地上下文能否回答问题：

- 现有实现模式。
- 包管理、框架和工具选择。
- 相似路由、组件、API、测试、迁移或配置。
- 代码或测试里已有的业务规则。

如果本地证据能回答问题，将其记录为带文件证据的推断澄清；只有后果有风险时，才向用户确认。

## 完成标准

最终需求摘要能说明以下内容时，澄清才算充分：

- 用户目标。
- 明确非目标。
- 期望行为。
- 重要边界情况。
- 验收标准。
- 现有项目约束。
- 仍保留的假设。
- 是否涉及 UI/E2E、API/数据、权限/auth、迁移或跨模块变更。
- 哪些回答改变了路由复杂度或风险。

## 给 Pilot 的输出

返回简洁交接内容：

```markdown
## 澄清后摘要

## 已确认决策
- 决策：影响

## 代码推断
- 事实：证据

## 剩余假设
- 假设：风险

## 路由信号
- 任务类型：
- 复杂度维度：
- UI/E2E:
- 专职 agent/skill 需求：
```
