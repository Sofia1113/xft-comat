# 测试先行开发（TDD）

## 概述

先写测试。看它失败。写最小代码让它通过。

**核心原则：如果你没有亲眼看着测试失败，你就不知道它测的是不是对的东西。**

**违背规则的字面，就是违背规则的精神。**

## 何时使用

**始终：** 新功能、缺陷修复、重构、行为变更。

**例外（需先征得用户同意）：** 一次性原型、生成的代码、配置文件。

冒出"就这一次跳过 TDD"的念头？停下。那是借口。

## 铁律

```
没有失败的测试，就不写产品代码
```

先写了实现再补测试？删掉，重来。

- 不要留作"参考"
- 不要一边写测试一边"改"它
- 不要看它
- 删就是删

从测试出发重新实现。没有例外。

## 红 - 绿 - 重构

### 红：写一个失败的测试

写一个最小测试，表达"应该发生什么"。一个行为、清晰命名、用真实代码（除非不得已才用 mock）。

✅ 好：

```typescript
test('失败操作重试 3 次后成功', async () => {
  let attempts = 0;
  const op = () => { attempts++; if (attempts < 3) throw new Error('fail'); return 'ok'; };
  const result = await retry(op);
  expect(result).toBe('ok');
  expect(attempts).toBe(3);
});
```

命名清晰、测真实行为、只测一件事。

❌ 差：

```typescript
test('重试能用', async () => {
  const mock = jest.fn().mockRejectedValueOnce(new Error())
    .mockRejectedValueOnce(new Error()).mockResolvedValueOnce('ok');
  await retry(mock);
  expect(mock).toHaveBeenCalledTimes(3);
});
```

命名含糊、测的是 mock 不是代码。

### 验证红：看它失败（强制，绝不跳过）

跑这个测试，确认：

- 是失败（fail），不是报错（error）
- 失败信息符合预期
- 失败是因为功能缺失，而不是拼写错误

测试直接通过了？你在测已有行为——修测试。测试报错？修掉错误，重跑直到它因正确原因失败。

### 绿：最小实现

写让测试通过所需的最简单代码。不加功能、不顺手重构、不"改进"超出测试要求的东西（YAGNI）。

### 验证绿：看它通过（强制）

跑测试，确认：本测试通过、其它测试仍通过、输出干净（无报错无告警）。测试失败就改代码不改测试；连带挂掉别的测试就现在修。

### 重构：清理

仅在绿之后：消除重复、改进命名、提取辅助函数。保持绿，不要加行为。

### 重复

为下一个行为写下一个失败测试。

## 并发实现中的单任务纪律

当你在并发批次里只负责一个任务时：

- 只做那一个行为（一个测试点 + 最小实现），只改动该任务声明的文件范围。
- 发现任务实际远超"一个测试点 + 最小实现"时，先上报拆分建议并停住，不自行扩范围。
- 只跑与本任务最相关的最小测试命令（单文件/单目录），不抢跑全量套件，避免与并发同伴互相干扰；全量回归留到最终验证。

## 缺陷修复中的 TDD

发现 bug？先写一个能复现它的失败测试，确认它现在失败，再按红-绿-重构修复。这个测试既证明修复有效，又防止回归。永远不要在没有测试的情况下修 bug。

## 好测试的标准

| 维度 | 好 | 差 |
|------|----|----|
| 最小 | 只测一件事；命名里有"和"就拆开 | `test('校验邮箱和域名和空格')` |
| 清晰 | 命名描述行为 | `test('test1')` |
| 表达意图 | 演示期望的 API | 掩盖代码应做什么 |

## 常见借口

| 借口 | 现实 |
|------|------|
| "太简单了不用测" | 简单代码也会坏，写测试只要 30 秒 |
| "我先实现，回头补测试" | 后补的测试一上来就通过，证明不了任何东西 |
| "我已经手动测过了" | 手动是临时的，没记录、不能重跑 |
| "删掉这几小时的代码太可惜" | 沉没成本；留着不可信的代码才是技术债 |
| "留作参考，再写测试" | 你一定会照着改，那就是后补测试。删就是删 |
| "测不动 = 设计不清" | 听测试的话：难测就是难用 |

## 红旗——停下，重来

- 测试之前先写了代码
- 实现之后才补测试
- 测试一写出来就通过
- 说不清测试为什么失败
- 测试"以后再加"
- "就这一次"

以上任何一条都意味着：删掉代码，用 TDD 重来。

## 完成前自检清单

- [ ] 每个新函数/方法都有测试
- [ ] 实现前亲眼看着每个测试失败
- [ ] 每个测试都因正确原因失败（功能缺失，而非拼写错误）
- [ ] 只写了让测试通过的最小代码
- [ ] 全部测试通过，输出干净（无报错无告警）
- [ ] 测试用真实代码（仅在不得已时用 mock）
- [ ] 覆盖了边界与错误场景

勾不全？说明你跳过了 TDD——重来。
