# 任务拆分方法论（task-splitting）

方案落定之后使用。目标：产出一份能让**多个执行者并发实现**的任务清单。无依赖且文件范围不相交的任务进同一波并发——拆分质量直接决定并发效率与合并安全。

**核心原则：每个任务一个可独立验证的行为，并发任务的文件范围互不相交。**

## 拆分规则

1. **任务单位 = 一个测试点 + 最小实现**：每个任务对应一个可独立验证的行为（一条测试用例 + 让它转绿的最小代码）。任务标题写明涉及文件。
2. **文件范围不相交是硬约束**：同时可执行（无依赖关系）的任务不得改动相同文件，也不得共享同一测试入口文件——这是并发合并安全的前提。范围重叠的任务用依赖串联成先后波次，而不是寄望执行者互相避让。
3. **按上下文边界定大小，不按钟表时间**：时长只是参考（单任务通常几分钟内可完成）。一个内聚单元（一个模块、一个组件、一组强耦合函数）拆碎后并发方仍要互相理解对方上下文时，**宁可保留为一个稍大的任务**——每个执行者冷启动都要重新读需求、设计与代码，切得过碎只会放大冷启动开销与协调成本。
4. **依赖显式化**：前置任务用依赖字段填任务 ID。典型顺序：数据结构/契约先行 → 各自独立的实现并行 → 跨模块集成收尾。
5. **自足性检查**：每个任务的标题 + 文件范围 + 设计文档，应足以让一个全新上下文的执行者独立完成，不需要追问其他任务的细节。做不到就说明拆错了边界。
6. **拆不动要说明**：确实无法拆开的大任务（如一次性的全局迁移），在设计文档写明原因与风险，不得为凑粒度硬切。

## 自检清单

- [ ] 同一波（无依赖关系）的任务文件范围两两不相交？
- [ ] 每个任务都有明确的验证方式（测试点）？
- [ ] 依赖链没有环、没有隐式顺序假设？
- [ ] 没有为凑粒度把内聚单元切碎？
- [ ] 拆不动的大任务有书面原因与风险？

## 示例（节选）

```text
TASK-001 角色-权限关联表与迁移（db/migrations/xxx.sql, models/role.ts）            deps: 无
TASK-002 权限校验中间件 + 越权 403 测试（api/middleware/permission.ts, tests/permission.spec.ts）  deps: TASK-001
TASK-003 角色列表 API + 测试（api/roles.ts, tests/roles.spec.ts）                  deps: TASK-001
TASK-004 前端权限勾选面板（web/components/RoleMatrix.tsx, web/__tests__/RoleMatrix.test.tsx）  deps: TASK-003
```

TASK-002 与 TASK-003 文件不相交且同依赖 TASK-001 → 同一波并发；TASK-004 依赖 API 形状 → 下一波。

拆出的每个任务由实现者按 [[tdd]] 测试先行完成。
