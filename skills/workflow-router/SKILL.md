# 工作流路由器（workflow-router）

前提：项目探索已给出项目快照，想法已澄清到可定级（用户确认过复述）。目标是选出唯一的工作流 mode 并写清理由。

**核心原则：路由判定由你（模型）做，先判类型，再评复杂度，安全/多模块/UI 命中底线绝不降档。**

## 判定步骤

1. **先判类型**（类型优先于复杂度评分）：
   - 用户描述错误、异常、失败测试、回归、线上问题或现有行为不符预期 → **bugfix**。
   - 用户要求整理结构、重命名、拆分、性能清理、复用、架构改善，且要求**外部行为不变** → **refactor**。
   - 其余是新增或改变功能 → 进入复杂度评分定 feature 档。

2. **复杂度自评**：对以下七个维度逐项判断是否命中，每命中一项计 1 分：
   - `requirements`：需求/边界/验收不明。
   - `scope`：≥3 文件或 ≥2 模块。
   - `infra`：API/DB/权限/异步/三方/计费/迁移/兼容。
   - `ui`：UI 流程/状态/可访问性/浏览器 E2E。
   - `decision`：架构取舍/方案比较/用户决策。
   - `risk`：安全/数据/回归/兼容/性能。
   - `test`：夹具/mock/浏览器/多轮验证。

   注意 `risk`：任务涉及认证/授权/密码/凭据/权限/支付/加密时本维**强制计 1 分**，不得以"演示项目/骨架/无生产数据"为由计 0——风险评的是该类代码出错的后果模式，不是当前数据量。

3. **定档**：`0-2 → feature-simple`，`3-4 → feature-medium`，`≥5 → feature-hard`。

4. **硬底线核对**：逐条核对安全底线，任一命中即取「评分档」与「底线档」的**较高者**作为最终 mode：
   - 新建/重构认证授权体系或安全敏感的分层结构 → 直接 `feature-hard`。
   - 涉及认证/会话/凭据/权限/支付/加密 → 最低 `feature-medium`。
   - ≥2 个新模块或用户要求拆分分层 → 最低 `feature-medium`。
   - 命中 `ui` 维度 → 最低 `feature-medium`。

   每条底线的命中/排除结论都要写进路由理由，不能只在心里过一遍。

5. **UI 标记**：若命中 `ui` 维度，必须显式标记本任务涉及 UI/E2E（收尾门禁据此强制 E2E 与前端设计决策不被静默降级）。UI 涉及面在更后阶段才浮现时，补开该标记。

## 路由理由应包含

- 选定 mode 与一句话定档结论。
- 任务类型判断（为什么是 bugfix/refactor/feature）。
- 复杂度逐维度命中情况与总分（feature 档）。
- 安全底线逐条命中/排除结论，以及底线是否抬高了评分档。
- 澄清如何改变或确认了类型、复杂度、风险、验收范围。
- 是否涉及 UI/E2E。

## 完成标准

- mode 唯一确定，且类型判断先于复杂度评分。
- 评分有逐维度依据，不是拍脑袋。
- 安全底线已逐条核对，命中安全/多模块/UI 底线的任务没有被降档。
- UI 任务已标记。
- 路由理由能让人复核"为什么是这个 mode"。

衔接 [[explore-project]] 的项目快照与 [[grill-idea]] 的想法还原；定档后由 [[solution-design]] 等进入设计。
