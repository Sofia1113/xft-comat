# 代码审查方法论

尽早审查，频繁审查——在问题级联放大之前抓住它。审查必须由独立的执行者完成，不能用"作者自检"替代：作者读的是自己的思路，审查者读的是产物。

**核心原则：对照需求看符合度，优先抓 correctness 与安全。**

## 审查顺序与清单

1. **符合度**：对照需求与设计以及用户的关键决策，确认实现确实满足需求、没有偏离已定方案。
2. **correctness（优先）**：边界条件、空值/异常、并发与顺序、错误处理、资源释放、off-by-one、状态一致性、回退路径。
3. **安全**：输入校验、注入、认证授权、敏感数据（密码必须哈希存储、token 策略明确）、越权、机密泄漏。发现安全风险时建议进入专门的安全审查，不在本轮替代它。
4. **回归**：本次改动是否破坏既有行为或相邻路径；重构任务确认外部行为等价。
5. **测试覆盖**：是否覆盖新增/变更逻辑、边界与错误场景；标出**必须补的回归测试**。
6. **可维护性（次要）**：命名、重复、复杂度、与既有模式一致性——不喧宾夺主。

## 发现格式

按严重度排序，每条含：`文件:行号` — 问题 — 影响 — 建议。明确区分两类：

- **阻塞（必须修）**：correctness 错误、安全漏洞、破坏既有行为、需求未满足。
- **建议（可后续）**：可维护性、风格、非关键优化。

## 红旗——不要这样审查

- 因为"看起来很简单"就跳过审查
- 只看风格、不看 correctness 与安全
- 报一堆主观偏好，淹没真正的阻塞问题
- 说"大概没问题"而没有逐条对照需求
- 把"作者自己跑过了"当作审查通过

## 闭环要求

- 审查结论要明确：**有没有阻塞发现**。有阻塞就附摘要，让后续修复有据可依；无阻塞就明说，修复环节据此跳过。
- 审查发现交回实现者闭环（补回归测试 + 修复 + 复跑），不要自己顺手改——审查者改代码就破坏了独立性。
- 指明修复后必须复跑的验证路径（自动化测试、E2E 路径）。
- 若发现影响设计，标注是否需要重新审查。

## 产出

- 审查结论（是否有阻塞）。
- 按严重度排序、带 `文件:行号` 的发现。
- 测试缺口与必补回归测试清单。
- 修复后必复跑路径，以及是否需重新审查。
- 残余风险。
