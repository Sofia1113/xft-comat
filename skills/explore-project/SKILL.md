# 项目探索方法论（explore-project）

在动手澄清或设计之前使用本 skill。目标：提供足够的项目上下文，避免在不了解代码现状时澄清或做技术决策。

**核心原则：先看清现状，再问问题、再做决策。只读，不动手。**

## 输入

- 用户原始请求。
- 当前工作目录。

## 方法

- **识别项目形态**：框架、语言、包管理器、测试工具、主要目录。优先看配置文件（package.json、pom.xml、Cargo.toml、Makefile 等）判断真实可用的构建/测试命令，而不是凭猜。
- **定位相关区域**：找到与用户请求最相关的代码、测试、配置或文档。
- **提取已有模式**：说明现有实现风格、相似功能、测试习惯和约束——后续设计应沿用既有模式，而不是另起一套。
- **发现风险信号**：指出是否涉及 UI/浏览器流程、API/数据、权限/认证、迁移、跨模块、风险热点。
- **给澄清输入**：列出代码无法回答、但会影响澄清或技术决策的问题。

## 边界

- **只读**：不修改任何文件，不执行耗时或破坏性命令。
- 不做最终技术决策，不替用户做产品决策——只提供事实与信号。

## 输出格式

```markdown
## 项目快照
- 技术栈：
- 包管理/测试命令：
- 相关文件：
- 现有测试：

## 现有模式
-

## 风险信号
- UI/E2E：
- API/数据：
- 权限/认证：
- 跨模块范围：
- 风险：

## 澄清输入
- 代码无法回答：
```
