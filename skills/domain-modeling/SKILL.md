# 领域建模方法论（DDD）

当任务的**业务规则复杂、概念多、边界不清或多模块协作**时使用。对纯技术性或单点改动的简单任务，不必强行套用，避免过度设计。

**核心原则：让设计围绕领域，而不是围绕技术分层。** 简单任务保持简单。

## 方法

1. **提炼通用语言（Ubiquitous Language）**：从需求与现有代码里抽出领域术语，确认每个词的确切含义，团队/代码/文档用同一套词。先探索代码已有命名，沿用而非新造。
2. **界定限界上下文（Bounded Context）**：识别本次改动落在哪个上下文，与其它上下文的边界与契约（谁拥有数据、跨边界如何通信、是否需要防腐层）。
3. **建模领域对象**：
   - **实体（Entity）**：有唯一标识、生命周期内可变。
   - **值对象（Value Object）**：无标识、按值相等、不可变（如金额、区间、坐标）。
   - **聚合（Aggregate）**：以聚合根为入口的一致性边界，外部只能引用根；定清不变量（invariant）。
   - **领域服务（Domain Service）**：不属于任何单一实体的领域行为。
4. **定不变量与规则归位**：把业务规则放进它该在的实体/聚合/服务里，避免贫血模型把规则散落到上层。
5. **映射到落地设计**：把领域模型映射成模块、接口、数据结构与改动点，把关键取舍列为待用户拍板的决策点。

## 边界

- 领域建模服务于设计清晰，不是为了引入框架或多加抽象层；简单任务保持简单（YAGNI）。
- 只产出设计，不写业务代码。
- 若涉及前端视觉/交互，明确需要前端设计参与的约束与验收点。

## 产出（并入设计文档）

- 通用语言术语表与限界上下文边界。
- 实体/值对象/聚合/领域服务的划分与不变量。
- 模块/接口/数据结构落地映射（衔接 [[solution-design]] 的推荐方案与 [[task-splitting]] 的任务拆分）。
- 需用户拍板的关键决策点。
