# Prompt 设计原则

提炼自 OpenAI《Prompt guidance》(GPT-5.4 / 5.5 与 GPT-5.3 Codex)。
原文：https://developers.openai.com/api/docs/guides/prompt-guidance

> **适用性说明**：原文面向 OpenAI 模型，本项目运行在 Claude 上。其中模型 / API 特定的内容（`phase` 参数、`apply_patch` 工具、`reasoning_effort` 与 `text.verbosity` API 参数、`gpt-5.4-mini/nano` 专项调法）在此**不收录**；本文只保留**模型无关、可迁移**的 prompt 设计模式，并标注它们在本项目 agent / skill prompt 中的落点。
>
> 本项目本身就是“以 prompt 为产品”——`skills/pilot/SKILL.md` 与 `agents/*.md` 都是 prompt。这些原则用于指导今后编写和修改这些 prompt，而非重复其中已有的职责与流程定义。

---

## 1. Outcome-first：给目标，别给步骤

- 优先描述**目标、成功标准、约束、可用上下文**，把“怎么走”留给模型，而不是把每一步钉死。模型会自行选择合适的搜索、工具或推理路径。
- **少用绝对词**（`ALWAYS` / `NEVER` / `must` / `only`）：只把它们留给真正的不变量——安全红线、必填输出字段、绝不能发生的动作。对“何时搜索 / 何时澄清 / 何时用工具 / 何时继续迭代”这类**判断题，改用 decision rule**（条件 → 动作），而不是命令式的步骤清单。

> **本项目落点**：对应“最小分派原则”——默认让主 Claude 描述目标自行推进，只在有独立判断价值时分派 specialist。agent 定义应描述“产出什么结论”，而非规定它内部每一步怎么做。

## 2. 明确“何时停”：stopping conditions 与预算

- 给出**停止条件**：每轮结束自问“现在能否用足够证据回答核心诉求？能就回答”。
- 用**最小充分证据**作答后即停；不要为润色措辞、补非必要细节而继续。
- 检索 / 工具循环要有**预算**：先做一次宽搜，够了就基于结果回答；只有在（核心问题未答、关键事实缺失、用户要求穷尽覆盖、必须读取某个指定文档、否则会产生重要无支撑断言）时才再检索。

> **本项目落点**：workflow 每个里程碑的收尾判断；`code-review` 与 E2E 阶段“够了就推进，不无限迭代”。

## 3. 输出契约与简洁度

- 用 **output contract** 固定输出：只返回要求的章节、按要求顺序；要求某格式（JSON / Markdown / SQL / XML）就只输出该格式；长度限制只作用于其对应章节，不要把 preamble 或分析块算成额外产出。
- **简洁但不残缺**：信息密度高、不复述用户请求、进度更新简短——但不要为了短而省掉必要证据、推理或完成检查。

> **本项目落点**：正是各 agent 现有的“**只输出 X / 不做 Y**”契约；延续这一写法，新 agent 也应显式声明输出边界。

## 4. 跟进策略与指令优先级：何时直接做，何时问

- **默认跟进策略**：意图清晰且下一步可逆、低风险 → 直接做；只有当下一步 (a) 不可逆、(b) 有外部副作用（发送 / 购买 / 删除 / 写生产）、(c) 需要缺失的敏感信息或会实质改变结果的抉择时，才停下来问。做完简述“做了什么、还剩什么可选项”。
- **指令优先级**：用户指令覆盖默认风格；安全 / 诚实 / 隐私 / 授权约束不让步；新指令与旧指令冲突时从新，不冲突的旧指令保留。
- **会话中途变更**：用 scoped 更新说清三件事——**Scope（作用范围）、Override（覆盖什么）、Carry forward（什么仍然有效）**。

> **本项目落点**：与 Claude Code harness 的既有行为同源（可逆直接做、不可逆先确认）；也契合本项目“`.xft-comat` 只能通过程序写、不得手动编辑”这类硬约束（属于不让步的 invariant）。

## 5. 工具使用：持续性、依赖、并行

- **持续性**：只要再调一次工具能实质提升正确性 / 完整性 / grounding 就别过早收尾；空或部分结果就换策略重试，直到任务完成且验证通过。
- **依赖检查**：动手前确认是否有前置发现 / 查询 / 检索步骤；**别因“终态显然”就跳过前置**；依赖上一步输出的，先解依赖。
- **并行 vs 顺序**：独立步骤并行以省时；有依赖、歧义或不可逆动作时顺序执行。并行检索后先汇总再继续，避免投机或冗余调用。

> **本项目落点**：conductor 编排时的阶段排序；TDD“先有失败测试再实现”的前置依赖；skill-scout 先 `skills list` 再判断。

## 6. 完整性与空结果恢复

- **完整性契约**：长任务 / 清单 / 批量 / 分页——任务视为未完成，直到所有要求项都被覆盖或显式标 `[blocked]`；维护内部清单，确认覆盖后再收尾；被缺失数据卡住的项标 `[blocked]` 并说明缺什么。
- **空结果恢复**：查询返回空、部分或异常窄时，**不要立刻断定“无结果”**；先试一两种回退（换措辞、放宽过滤、补前置查询、换源 / 换工具），再报告并说明试过什么。

> **本项目落点**：hard 流程的需求 / 测试覆盖完整性；E2E 用例逐条核对、未过项打回重测。

## 7. 高影响动作前的验证闭环

- 任务看似完成后，加一道轻量**验证**再返回或执行不可逆动作：核对正确性（满足每条要求？）、grounding（事实有上下文 / 工具输出支撑？）、格式（符合 schema / 风格？）、安全与不可逆性（有外部副作用就先征求许可）。
- **缺上下文门控**：必需上下文缺失时不要猜；可检索就用工具取，不可检索才问一个最小澄清问题；不得不推进时显式标注假设并选可逆动作。
- **动作安全帧**：pre-flight（一两行说明将执行的动作和参数）→ 执行 → post-flight（确认结果与已做的验证）。

> **本项目落点**：本项目“**代码审查早于最终 E2E 验证**”正是这一思想；close 前的 `validate` 检查是收尾验证闭环。

## 8. 人格与协作风格分离

- **人格**（怎么说话：语气、温度、直接程度、正式度、幽默、共情）与**协作风格**（怎么干活：何时问、何时假设、多主动、何时检查工作、如何处理不确定与风险）是两件事，分开写、都保持简短。
- 两者都**不能替代**清晰的目标、成功标准、工具规则和停止条件。

> **本项目落点**：agent 角色定义（如 conductor 的“指挥而不亲自实现”）属协作风格；不要用人格描述去掩盖缺失的职责契约。

## 9. 可见前导（Preamble）

- 多步或工具密集的任务，在开始动作前先发一条**简短可见更新**（一两句：确认请求 + 说明第一步），提升“首字节”响应的感知速度，不改变底层任务。

> **本项目落点**：长 workflow 推进时，先给用户一句“采用哪种模式、第一步做什么”，再开干。

## 10. Grounding、引用与创作护栏

- 把**引用行为写进 prompt**：什么需要支撑、多少证据算够、证据缺失时怎么办。**证据缺失 ≠ 自动判定为“否”**。
- **创作护栏**：起草幻灯片 / 对外文案 / 摘要等时，区分“必须来自来源的事实”与“可创造性书写的部分”；不要为了更有说服力而编造具体名称、数据、路线图状态或产品能力；缺乏可引用支撑时，写带占位符或明确标注假设的通用稿，而非杜撰具体内容。

> **本项目落点**：架构方案、根因分析等必须基于已确认需求 / 证据；无证据处显式标注假设而非杜撰。

## 11. 落地方法论：effort 是 last-mile，改动一次一个

- **reasoning effort 是最后一公里的旋钮**，不是提质的主路径。先靠更好的 prompt（output contract、completeness、verification loop、tool persistence）拿回大部分性能，再考虑调 effort。
- **一次只改一个变量**：从能通过 eval 的**最小 prompt** 起步，只在**修复实测到的失败模式**时才加一个 block；不要预防性堆砌规则。
- 若模型太字面化、止步于第一个看似可行的答案，先加一句“别停在第一个答案，找二阶问题、边界情况、缺失约束；关键任务至少做一次验证”，再考虑提 effort。

> **本项目落点**：维护本插件的 SKILL.md / agents 时同理——先用最小 prompt 加明确契约，遇到实测失败再补 block，避免规则膨胀与冗余文档。

## 建议的 prompt 结构（复杂 prompt 起点）

每节都保持简短，只在会改变行为处加细节：

```
Role:               1-2 句，定义模型的功能、上下文、职责
# Personality       语气、姿态、协作风格
# Goal              用户可见的目标产出
# Success criteria  最终答案前必须为真的条件
# Constraints       policy / 安全 / 业务 / 证据 / 副作用限制
# Output            章节、长度、语气
# Stop rules        何时重试、回退、弃答、询问或停止
```

## 写 / 改 agent prompt 时的对照清单

- [ ] 写的是目标与成功标准，而非僵硬的步骤清单？
- [ ] 绝对词只用在真正的不变量上，判断题改成了 decision rule？
- [ ] 有明确的停止条件 / 完成定义？
- [ ] 输出契约清晰（章节、顺序、格式、长度边界）？
- [ ] 何时直接做、何时停下来问，有可逆性 / 副作用判据？
- [ ] 工具有依赖检查与持续性规则，空结果有回退策略？
- [ ] 高影响 / 不可逆动作前有验证闭环？
- [ ] 人格与协作风格分开且简短，没用来掩盖缺失的职责契约？
- [ ] 是最小可用 prompt，新增的每个 block 都对应一个实测失败？
