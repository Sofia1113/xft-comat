工作流需求如下
  1. 首先最重要的一点是当前工作流需要极致的轻量化，要求可以按照预设定的工作流快速的有序、有目的的执行。
  2. 要有5种工作流模式，三种工作流对应简单、中等、困难级别的新需求任务，bug修复以及重构任务，每种任务都需要有特定且正确的工作流方式。
  3. 工作流改执行那种需要智能的路由，要有理有据，路由到对应的工作流需要一定的标准衡量。
  4. 需要有文档留存记录，维护一个目录，目录下每个任务都有对应的文件夹，文件夹的命名日期-topic，目录下存放工作流对应的文档，比如困难的新需求任务，
    a. 用户需求（澄清之后产出）
    b. 设计方案（通过架构师agent产出，用户抉择在交互完成之后添加到设计方案的前方）
    c. 测试用例（在E2E的验证阶段会对照着测试用例，测试通过的用例需要打勾，
    测试未通过的情况需要打回继续修复问题，再次测试时需要重新生成一个测试用例文档进行打勾标记通过，多少轮就保留多少个文档）
    d. 状态机文件（极简，在工作流敲定之后生成一个对应工作流的流程步骤内容，不关注该阶段输出什么输入什么，只需要看到整个工作流有哪些阶段，执行到了哪些阶段）
    e. 实现的任务列表（任务的拆解以及任务的完成情况）6、合理的赋能的skill介入帮助时的记录文件，简单记录了使用了哪些skill 。
  其他的工作流也需要有对应其流程应该产出的文档，每个文档都必须要有一个模板文件，不能脱离模板乱写。模板文件需要贴合场景，必须是非常合理的。
  5. 上述目录只能通过程序去维护，不要直接操作该目录下的文件只能读取。
  6. 工作流的设定需要固定在程序中，对应工作流需要生成对应的文档这个是固定的。
  7. 执行层采用「通用 worker + 专业 skill」模型：只保留两个通用执行 agent（worker 读写 / worker-ro 只读，对应工具级权限隔离），它们熟知 workflow 脚本协议，通过脚本获取当前阶段的分派包并装载该阶段的方法论 skill 完成任务。每个阶段的专业能力（探索、调查、设计、拆分、TDD、审查、E2E）收敛在各自的 SKILL.md 中，skill 只关注领域方法论，不关心工作流噪声。
  8. worker 的设定只有执行协议（取分派包 → 读 skill → 干活 → 自录 → 汇报），不出现任何领域设定；阶段序列与"下一步做什么"由 workflow 脚本（状态机）负责；指挥者是 pilot 主会话，它也只按脚本返回的分派包合理推进任务，advance/close 只归主会话。
  9. 对于实现阶段的 worker 来说，TDD（tdd skill）是保证代码质量的必要因素。
  10. 代码审查由独立分派的只读 worker 装载 code-review skill 负责，该 skill 需要参照 claude 官方插件中的代码审查；审查者与实现者必须是不同的执行上下文。
  11. 工作流在一些阶段需要支持调用用户自备的skill，在特定的软件开发环境下这个十分关键。可以在工作流的的实现阶段开始之前，通过获取 skill 的程序去读取现有的 skill，分析出合适调用的 skill 询问用户是否使用。
  12. 需求的澄清需要类似拷问的方式，不要一股脑的问出一堆问题之后就结束了，每个回答都可能会产生新的问题，需要持续的收集用户的需求，直到最终按照用户想法可以没有任何疑问的产出符合用户想法的设计文档，最终在用户确认一致之后进行存档。
  13. E2E 的验证不是单纯的跑单元测试，如果有UI的内容需要有浏览器介入的集成测试，这在测试用例的生成时是关键要素。浏览器的能力就使用 agent-browser 这个 skill，这个 skill 就作为当前项目的一个固定必备 skill。


强调一点非常重要的事，在之前的几个版本测试过程中，我发现了工作流本身对于 AI Coding 来说是非常限制手脚的一个东西，但是没有规范的工作流AI 又像脱缰的野马无法掌控，但是在使用工作流的情况下测试让 AI 完成一些开发任务，AI 超过百分之 50 的时候都在服务与工作流而非专注在任务实现上，例如每个 agent 指派之后都需要将 .orbit 目录下的内容全部通读一遍，然后还要更新里面属于自己的部分，最后才开始完成任务，任务好不容易跑完之后，还需要维护结果和状态。这太耗费时间和 token 了。变得极其笨重的不灵活，成本也极其高昂。我希望的工作流是可以像程序一样去执行，设定好就只管执行，在需要上下文的情况下才去收集。就像需求的第一点，最重要的事情就是保证极致的轻量和快速，但是仍要保证按照工作流正确执行任务。



# xft-comat 工作流用户需求

本文档从用户视角描述 xft-comat 工作流应满足的行为要求，用于约束 skill、agent 编排和评测标准。用户期望的不是形式上生成一组 `.xft-comat` 文件，而是让工作流真正改善复杂任务的澄清、设计、实现、审查和交付质量。

## 用户目标

- 用户希望 xft-comat 在复杂 AI Coding 任务中承担“流程指挥者”的角色，能够组织澄清、方案、实现、测试、审查和收尾，而不是只在主会话里凭模型自觉完成所有工作。
- 用户希望 hard 流程能体现多执行上下文深度协作：方案设计、TDD 实现、E2E 验证、代码审查应由独立分派的 worker 装载对应方法论 skill 在合适阶段实际产出判断或工作结果；需求澄清由主会话主持，保持低延迟交互。
- 用户希望每个阶段的产物足以复盘：为什么这样定级、问过哪些需求、哪个执行者完成了设计或审查、测试和审查发现了什么、最终还有哪些风险。

## 工作流判断要求

- 流程开始时不能只凭用户首条任务立即做最终工作流定级；对于需求明显模糊、范围可能影响复杂度或风险等级的任务，应先做最小必要的需求澄清，再确认或修正 `feature-simple`、`feature-medium`、`feature-hard`、`bugfix`、`refactor` 等模式。
- 允许先做脑内临时路由假设以决定澄清深度，但对话一开始不应先执行 `workflowctl.ts route`；`route` 应在公共的接收、轻量探索、需求澄清之后，用澄清后的任务摘要执行。
- 五种工作流应共享前置公共阶段（接收任务、探索与澄清），最终路由位于公共阶段之后，再进入各 mode 专属阶段，避免一开始让用户等待路由脚本。
- 五种工作流的阶段必须来自同一组原子阶段；同一语义只能有一个阶段名，不能因为不同 workflow 把相近动作命名为 `targeted-test`、`baseline-test`、`regression-verify`、`verify-equivalence` 等分叉阶段。
- 该假设必须标记为 provisional，并在澄清后写入最终路由结论。
- 路由记录必须说明哪些澄清答案影响了最终复杂度评分，避免出现“未澄清就已经定死 hard/medium/simple”的审计断层。

## 想法澄清要求（grill-idea，替代清单式需求澄清）

- 路由前的澄清由主会话主持，调用 `grill-idea` skill（源自 grill-me 模式），目标是**还原用户想法**并达成可复述的共同理解，而不是填满需求清单：理解用户为什么要做、想得到什么、心中已有的画面、没说出口的假设。
- 访谈机制：**一次只问一个问题**，回答决定下一个问题；每个问题附带推荐答案和理由；沿决策树按依赖顺序逐个解决（动机 → 期望成果 → 形态与行为 → 边界与例外 → 验收 → 路由信号）；与任务无关的分支明确跳过，不为问而问。
- 优先使用当前运行时提供的提问工具（如 `AskUserQuestion`、`request_user_input` 或等价工具），推荐答案放第一个选项；只有工具不可用时才降级为普通文本提问。
- 能通过探索代码库回答的问题必须探索代码库，不转问用户；这类结论记录为带文件证据的推断，只有后果有风险时才向用户确认。
- 澄清的完成标准是共同理解：向用户复述其想法（要什么、为什么、长什么样、不做什么、怎么算成）并获用户确认。对 hard 流程，数据库类型、权限边界、数据安全、UI 完整度、导出规模、部署方式、验收方式等仍不确定时，必须继续追问或明确记录为待确认假设。
- `01-requirements.md` 必须保留逐问澄清记录，说明每个问题为什么重要、推荐答案是什么、用户如何回答、哪些答案改变了范围、设计或路由复杂度。
- 澄清由主会话直接主持，保持低延迟问答，不绕 subagent 来回转述；如需独立视角找盲点，可让 worker-ro 出一份缺口清单或在收尾时审一次需求归档。

## Skill 参与要求

- 当任务明确包含前端界面、视觉风格或交互体验时，`frontend-design` 不能只被加载或记录为 required；它必须实际参与界面方向、布局、控件、视觉规范或可用性检查，并将结论反映到设计文档或实现约束中。
- 当任务包含 UI/E2E 时，`agent-browser` 必须执行真实浏览器验证，并把关键验证路径、失败、修复和最终结果记录到测试文档。
- 当流程记录某个 skill 为 required 时，必须能在会话记录中找到该 skill 的实质输出或操作证据；否则应记录为未实际参与的偏差。

## Worker 编排要求

- 主会话是工作流编排者，不是文件修改者：主会话不得直接修改任何项目文件、代码文件、测试文件、配置文件或 `.xft-comat` 文档；唯一允许产生文件变更的动作，是调用插件内 `workflowctl.ts` 维护 `.xft-comat/`。
- 业务代码和测试变更必须由被分派的 worker 或明确的外部 skill 执行；主会话只做澄清、路由判断、worker 分派、证据核验和最终汇报。
- hard 流程必须体现 conductor/指挥者职责：循环向脚本要分派包、分派对应 worker 变体、汇总 worker 输出、决定是否 advance，并指出哪些结论应由主会话通过 `workflowctl.ts` 写入工作流文档（仅路由两文档与 record-decision）。
- hard 流程不应主要由主会话独自完成实现、验证和审查；主会话可以协调和整合，但关键阶段（plan/implement/review）必须有独立分派的 worker 深度参与并留下阶段执行记录。
- plan 阶段的 worker 应装载 solution-design（必要时 domain-modeling）参与方案比较和取舍、按 task-splitting 拆分任务；implement/fix 的 worker 应装载 tdd 做测试优先实现；final-verify 的 worker 应装载 e2e-verification 执行端到端验证；review 的 worker 应装载 code-review 完成审查。
- 如果某个阶段没有由独立 worker 执行（主会话兜底或跳过），必须在阶段执行记录中以 executor=main + 原因或 skipped + 原因显式留痕，例如运行时无 subagent、分派失败、任务降级或该阶段风险较低。

## 审查与验证顺序要求

- hard 流程中，代码审查不应放在所有测试验证之后才首次出现。合理顺序应至少包括：实现后先运行必要的自动化测试确认可审查状态，然后进入代码审查，再根据审查意见补测试和修复，最后执行完整验证和 E2E 回归。
- E2E 验证用于确认审查修复后的用户可见行为，不能替代代码审查。
- 若先做了完整 E2E 后才发现代码审查问题，必须重新执行受影响路径的自动化测试和 E2E，并在测试记录中说明这是审查后的回归验证。

## 代码审查要求

- 代码审查阶段必须使用 xft-comat 的代码审查能力：由独立分派的 `worker-ro` 装载插件的 `code-review` skill 完成；本机存在更合适的通用 review skill 时也应明确调用，而不是只让主会话自检。
- 审查结果必须按严重程度记录 correctness、安全、回归风险、测试缺口和可维护性问题。
- 审查发现的问题必须形成闭环：补回归测试、修复、复跑测试、必要时重新 E2E，并把关闭状态写入工作流文档。

## 验收标准

- 用户可以从 `.xft-comat` 目录独立复盘完整流程，包括澄清、最终路由、架构选择、各阶段执行记录（哪个执行者 + 什么证据）、skill 实际参与、测试与审查闭环。
- hard 流程的 `workflow.json` 最终必须显示 `close` completed，不能停留在 `in_progress`。
- `05-tasks.md` 不能保留模板占位；必须反映实际任务拆分、owner（worker）、完成状态。
- `06-skill-usage.md` 不能同时保留“无，待记录”和 required skill 记录；required skill 必须有实质参与证据。
- 最终回复中的“已完成流程”必须与工作流文件状态一致，不能口头声称已 close 但记录仍处于进行中。
